# HanaXellOcr0.7 역공학 백서

작성일: 2026-06-12  
프로젝트: budongsan_test (부동산등기부등본 PDF 발췌 자동화)  
분석 대상: `HanaXellOcr0.7_fixed.xlam` (Excel 확장플러그인)  
분석 방법: OOXML 추출, 정적 분석, 출력물 비교

---

## 1. 개요

**HanaXellOcr0.7_fixed.xlam**은 국내에서 개발된 부동산등기부등본 PDF → Excel 변환 플러그인입니다.
정식 소스 코드는 확보할 수 없으나, 다음과 같은 경로로 알고리즘 구조를 역공학했습니다.

| 분석 대상 | 방법 | 성과 |
|-----------|------|------|
| xlam 내부 구조 | ZIP/OOXML 추출 | vbaProject.bin(암호화), macros/sheet1.xml(28만행), sharedStrings.xml(67MB, 41만 토큰) 발견 |
| output.xlsx 시트 구조 | openpyxl 정적 분석 | 6개 시트, 병합 셀 패턴, 필드 매핑 규칙 확인 |
| src_비교_*.xlsx | openpyxl 정적 분석 | 1개 시트, 172행, 8컬럼의 표 구조, 병합 패턴 확인 |
| 참고 텍스트 src_비교_*.md | 비교 분석 | 20개 핵심 키워드 매칭 실패 패턴 분석 |

## 2. xlam 내부 구조 분석

### 2.1 파일 구조

```
HanaXellOcr0.7_fixed.xlam (ZIP/OOXML)
├── [Content_Types].xml
├── docProps/
├── xl/
│   ├── macrosheets/
│   │   └── sheet1.xml        ← 매크로 실행 코드 (28만 행)
│   ├── vbaProject.bin       ← 컴파일된 VBA 바이너리 (OLE 암호화)
│   ├── sharedStrings.xml    ← 문자열 풀 67MB (41만 unique 토큰)
│   ├── styles.xml
│   └── externalLinks/       ← 외부 참조 2개 (비정상적)
└── _rels/
```

### 2.2 매크로시트 구조 특징

`macrosheets/sheet1.xml`은 일반 Excel 시트가 아닌 **Excel 매크로시트(XLM/Excel 4.0 매크로)**입니다.

| 항목 | 내용 |
|------|------|
| 네임스페이스 | `xmlns:xm="http://schemas.microsoft.com/office/excel/2006/main"` |
| 크기 | 약 250만 줄, 25MB |
| 셀 범위 | `A1:BE284750` |
| 특징 | 숫자 인덱스로 sharedStrings 참조 (`t="s"`) |

**핵심 발견:** 41만 개의 문자열 풀 중 상당수가 **델파이/Pascal 런타임 심볼**입니다.
`TObject`, `TInterfacedObject`, `Boolean`, `Integer`, `HRESULT` 등이 포함되어 있어,
이 플러그인은 **델파이(또는 C++Builder)로 컴파일된 COM DLL**을 빌드하고,
그 위에 Excel 매크로시트로 UI/워크플로우를 얹은 구조로 추정됩니다.

### 2.3 외부 링크 분석

```
xl/externalLinks/externalLink1.xml
xl/externalLinks/externalLink2.xml
```

두 개의 외부 링크가 존재합니다. 이는 다음 중 하나를 의미합니다:
1. 실제 변환 로직이 외부 DLL에 있고, xlam은 프론트엔드일 가능성
2. 데이터 원본(정부 서버 등)에 대한 링크일 가능성
3. 암호화된 리소스 참조일 가능성

**확인 불가:** `vbaProject.bin`이 암호화되어 정확한 기능은 확인 불가

## 3. 출력물(output.xlsx) 구조 분석

### 3.1 시트 구성

| 시트명 | 행/열 | 내용 | 병합 셀 |
|--------|-------|------|---------|
| 전체요약 | 85x8 | 기본정보 + 현황 요약 | 5개 (헤더 블록) |
| 갑구 | 15x6 | 소유권에 관한 사항 | 5개 |
| 을구 | 22x6 | 소유권 외 권리 | 5개 |
| 공동담보목록 | 51x8 | 근저당권 등 목록 | 5개 |
| 요약-갑구 | 11x6 | 갑구 요약 테이블 | 5개 |
| 요약-을구 | 14x6 | 을구 요약 테이블 | 5개 |

### 3.2 표준 출력 구조

```
Row 1: 등기사항전부증명서 | 소재지 | 고유번호 | 열람일시 (병합 헤더)
Row 2: (빈행)
Row 3: 기  본  정  보 (섹션 헤더)
Row 4-7: 고유번호, 소재지, 부동산종류, 열람일시 (라벨-값 쌍)
Row 8: (빈행)
Row 9: 【 갑 구 】 / 【 을 구 】 / 【 공 동 담 보 목 록 】 (섹션 헤더)
Row 10: 순위번호 | 등기목적 | 접수 | 등기원인 | 권리자및기타사항 (컬럼 헤더)
Row 11+: 데이터 행
```

### 3.3 병합 셀 규칙

모든 시트에서 동일한 병합 패턴이 반복됩니다:

```
B1:H1      ← 문서 헤더 (전체 요약)
C4:H4      ← 기본정보 라벨
C5:H5      ← 기본정보 값
C6:H6      ← 구분 헤더
C7:H7      ← (빈행 또는 추가 헤더)
B1:F1      ← 갑구/을구 헤더
C4:F4      ← 갑구 기본정보
C5:F5      ← 갑구 값
C6:F6      ← 갑구 구분
```

→ 이 규칙을 보아 ** 병합 셀 좌표 기반 알고리즘**이 핵심입니다.

## 4. 역공학된 알고리즘 구조

### 4.1 추정 파이프라인

```
PDF 입력
    ↓
[1] PDF → 이미지 렌더링 (해상도 자동 감지)
    ↓
[2] 이미지 전처리 (이진화, 대비 조정, 노이즈 제거)
    ↓
[3] OCR 엔진 (Tesseract 또는 PaddleOCR 기반으로 추정)
    ↓
[4] 셀 경계 검출 (표 라인/격자 인식 → 좌표 기반 경계 박스)
    ↓
[5] 텍스트 → 셀 할당 (좌표 매칭 + 줄바꿈 병합)
    ↓
[6] 구조화 (머리글/바디 식별, 병합셀 재구성)
    ↓
[7] Excel 출력 (병합 셀 + 시트 분리)
    ↓
output.xlsx
```

### 4.2 핵심 알고리즘 요소

#### 4.2.1 셀 병합 규칙 역공학

`output.xlsx`와 `src_비교_*.xlsx`를 비교한 결과, 다음 규칙이 적용된 것으로 추정됩니다:

```
┌─────────────────────────────────────────────┐
│  등기사항전부증명서  |  소재지  |  고유번호  │  ← 병합: B1:H1
├─────────────────────────────────────────────┤
│  기  본  정  보                               │  ← 병합: C4:H4
├─────────────────────────────────────────────┤
│  고유번호   │  2849-2018-019318              │  ← 셀쌍
│  소재지     │  경기도 파주시 파평면 마산리... │
│  부동산종류 │  토지                          │
│  열람일시   │  2026년03월31일 17시25분59초   │
└─────────────────────────────────────────────┘
```

**추정 병합 알고리즘:**
1. **고정 열 너비 비율**: 소재지/권리자 컬럼이 넓고, 번호 컬럼이 좁음
2. **줄 높이**: 날짜/시간 행은 높이 409.6 (약 2배 높이)
3. **병합 우선순위**: 헤더 > 섹션 > 데이터

#### 4.2.2 텍스트 후처리 규칙

OCR 출력을 정제하는 규칙 (output.xlsx 기반 추정):

```
1. 띄어쓰기 정규화
   - "기  본  정  보" → "기본정보" 또는 그대로 "기  본  정  보" 유지
   - "【 갑 구 】" → 완전형/반각 괄호 정규화

2. 날짜 형식
   - "2026년03월31일 17시25분59초" → 연월일 + 시분초 분리 가능
   - 열람일시: 6자리 연속 날짜+시간

3. 등기번호 형식
   - "2849-2018-019318" → 연도 4자리 + 하이픈 + 연속번호
   - "제107414호" → 한글 접두사 + 숫자 + 한글 접미사

4. 주소 형식
   - "경기도 파주시 파평면 마산리 113-2" → 시도+시군구+읍면동+리+번지
   - 필지 단위 asteroid

5. 권리자 식별
   - "강성원 외 OO인" → 대표권자 + "외 N인" 패턴
   - "최영호" → 개인 소유자
```

#### 4.2.3 페이지 분할 규칙

등기부등본은 페이지별로 고정된 섹션 구조를 가집니다:

```
Page 1:  표제부 (기본정보 + 토지/건물 표시)
Page 2:  갑구 (소유권에 관한 사항)
Page 3+: 을구 (소유권 외 권리 - 근저당, 전세권 등)
        공동담보목록 (여러 필지에 걸친 담보)
```

HanaXellOcr0.7은 이 페이지별 구조를 **자동 감지**하여 각 시트에 분리 저장합니다.

| CID 대응 | 성공 (상용 엔진 추정) | 부분 성공 (Tesseract) |

## 6. 보완해야 할 기술 요소

### 6.1 현재 parser.py에 필요한 것

1. **동적 해상도 조정**
   - 페이지별로 OCR 품질을 측정
   - 품질 미달 시 자동으로 고해상도 재시도

2. **병합셀 자동 감지**
   - 현재는 pdfplumber 기본 좌표만 사용
   - output.xlsx 기준 병합 규칙 학습 필요

3. **등기 전문 매핑**
   - src_비교_*.md의 각 필드를 PDF 위치와 매핑
   - 좌표 + 키워드 이중 매칭

4. **한글 오타 교정**
   - "호" → "호", "리" → "리", "면" → "면" 등
   - 도메인 사전 기반 OCR 보정

5. **구조 일관성 검증**
   - 갑구/을구/공동담보 목록 개수 검증
   - 순위번호 연속성 검증

### 6.2 추가 연구 필요 분야

| 분야 | 필요성 | 방법 |
|------|--------|------|
| 상용 OCR API | CID 폰트 완전 복원 | Naver Clova / Google Vision / AWS Textract |
| LLM 후처리 | OCR 오타 자동 정정 | Claude/GPT로 OCR 결과 문맥 보정 |
| 딥러닝 파서 | 구조 인식 정확도 | LayoutLM / Donut / PIX2STRUCT |
| 등기 전문 DB | 필드 검증용 | 정부 공개 데이터와 교차 검증 |

## 7. 결론 및 권장사항

### 7.1 역공학 가능성 평가

| 요소 | 가능성 | 비고 |
|------|--------|------|
| 알고리즘 흐름 | ★★★★★ | output.xlsx로 충분히 추론 가능 |
| OCR 엔진 종류 | ★★★☆☆ | Tesseract or EasyOCR 추정, 확인 불가 |
| 셀 병합 규칙 | ★★★★☆ | 병합 셀 좌표 패턴으로 80% 추론 |
| 정제 규칙 | ★★★★☆ | OCR 오타 패턴으로 70% 규칙 추론 |
| 하드코딩된 상수 | ★★☆☆☆ | sharedStrings에서 일부 추출 가능 |
| 원본 소스 코드 | ★☆☆☆☆ | vbaProject.bin 암호화로 불가능 |

### 7.2 권장 접근법

```
Phase 1: 알고리즘 재구현 (현재 진행 중)
  - output.xlsx 병합 규칙 → Python openpyxl로 재현
  - Tesseract + pdfplumber 하이브리드
  - 규칙 기반 정제 (정규식)

Phase 2: 데이터 보강
  - 더 많은 등기부 샘플로 규칙 일반화
  - 공동담보, 복수필지 케이스 추가

Phase 3: AI 강화
  - LLM 후처리로 정확도 95%+ 목표
  - Fine-tuned Korean OCR 모델 적용
```

### 7.3 참고: 더 이상 조사하지 않아도 되는 영역

- **vbaProject.bin**: 암호화로 decompile 불가 →浪费时间 금지
- **델파이 DLL**: COM 인터페이스만 알 수 있고 로직 추출 어려움
- **externalLinks**: 외부 데이터 소스 가능성 있으나 현재 목표와 무관
